## 1. 项目概述

### 1.1 项目目标
本项目旨在构建一个可扩展、高性能的NBA数据平台，支持：
- 多支球队历史比赛数据（以马刺队为主，逐步扩展）
- 传奇球员完整职业生涯数据（支持跨球队、跨年龄分析）
- 球队阵容、交易、高阶数据分析
- 前端可视化展示（战绩、年龄曲线、球员对比等）

### 1.2 技术栈
- **后端**: FastAPI + Python 3.11+
- **数据库**: PostgreSQL（推荐15+）
- **前端**: HTML + JavaScript + Chart.js（可后续升级 React/Vue）
- **数据处理**: pandas + SQLAlchemy
- **配置管理**: python-dotenv

## 2. 数据库设计（核心改进）

### 2.1 设计原则
- **规范化 + 维度建模**，避免按赛季/球员建表导致表爆炸
- 支持**任意球员跨球队、跨年龄、跨赛季**灵活查询
- 区分**明细数据**与**汇总数据**，提升查询性能
- 所有高阶指标在ETL阶段计算并存储

### 2.2 核心表结构

**维度表**：
- `players` — 球员基础信息
- `teams` — 球队信息
- `seasons` — 赛季维度（含常规赛/季后赛）

**事实表**（核心）：
- `player_team_seasons` — 球员-球队-赛季汇总（强烈推荐重点使用）
- `player_game_logs` — 球员逐场比赛明细
- `team_games` — 球队逐场比赛数据（类似你之前的湖人CSV）

**详细DDL请参考** `docs/database_schema.sql`（后续可生成）。

### 2.3 表命名规范（新）
- 使用**小写 + 下划线**（snake_case）
- 避免按赛季动态建表
- 示例：`player_team_seasons`、`spurs_2025_26_games`

## 3. 数据获取原则（强制规范）

### 3.1 核心原则
**所有前端展示的数据必须通过API从数据库获取，禁止直接读取JSON文件作为数据源。**

### 3.2 数据源优先级
1. **PostgreSQL数据库**（主数据源）
2. **SQLite数据库**（备用数据源）
3. **CSV文件**（仅用于数据导入）

### 3.3 规范要求
- ✅ 所有API接口必须从PostgreSQL数据库获取数据
- ✅ 禁止在API代码中直接读取JSON文件作为主要数据源
- ✅ CSV文件仅作为原始数据导入源，不应被前端直接访问
- ✅ 数据变更通过ETL流程同步到数据库，再由API提供给前端

### 3.4 赛季格式规范
- URL参数格式: `2025-26`（单年份格式）
- 数据库存储格式: `2025-2026`（完整赛季格式）
- API需要自动转换: `normalize_season()` 函数

### 3.5 球队ID规范
- `teams.team_id`: 数字ID (如 `2008`)
- `teams.team_abbr`: 球队缩写 (如 `LAL`, `SAS`)
- `team_games.team_id`: 使用 `team_abbr` (如 `LAL`, `SAS`)
- API同时支持数字ID和缩写查询
